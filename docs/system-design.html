1. 執行摘要

AiraHost 為 Airbnb 房東提供逐日定價建議。使用者描述房源（或提供 URL）後，Python Worker 爬取 Airbnb 尋找相似房源，計算相似度加權市場中位數，並按日疊加需求與時間調整，可選擇以使用者指定的基準房源錨定結果。

系統有兩類報表：即時報表（使用者觸發、自訂日期範圍、出現於歷史記錄）與夜間報表（排程器觸發、固定 30 天、成為儀表板看板的真實資料來源）。兩者共用同一執行引擎（_execute_analysis），但抓取預算與快取規則不同。

正式的使用者端定價欄位為 CalendarDay.recommendedDailyPrice。日曆中所有其他定價欄位均為舊版相容別名或內部流水線階段。本文件精確說明整個系統，並全程引用程式碼路徑。

2. 系統總覽

高階架構圖

系統有三個執行時角色：

  • Next.js App Router — 提供前端與所有 /api/* 路由。以 Zod 驗證輸入，將 pricing_reports 列寫入 Supabase，並將結果傳回瀏覽器。
  • Supabase（PostgreSQL） — 持久化儲存使用者、已儲存房源、定價報表、房源–報表連結及觀察資料。Row-Level Security 確保使用者資料隔離。
  • Python Worker — 部署在 Railway 的長期執行程序。輪詢 Supabase 取得 queued 報表，認領後以 Playwright CDP 爬取 Airbnb、執行定價並將結果寫回。無 HTTP 伺服器 — 純輪詢處理迴圈。

Railway cron 每日呼叫 POST /api/internal/nightly/schedule 一次，為所有符合條件的已儲存房源排入夜間看板更新報表。

來源：docs/PROJECT_MEMORY.md、worker/main.py、src/app/api/internal/nightly/schedule/route.ts

5. 執行政策模型

Phase 1 後建立的每一個 pricing_reports 列均帶有明確的 result_core_debug.execution_policy 欄位。Worker 讀取此欄位以選擇流水線。舊版列（Phase 1 前）則從 job_lane 推導政策作為備援。

政策對照表

政策狀態job_lane觸發方式流水線
nightly_board_refresh 啟用中 nightly scheduled run_nightly_job()

解析演算法（worker/core/report_policy.py）

def resolve_execution_policy(job):
    # 1. 明確欄位：從 result_core_debug 讀取（Phase 1 報表）
    debug = job.get("result_core_debug")
    if isinstance(debug, dict):
        explicit = debug.get("execution_policy")
        if isinstance(explicit, str) and explicit in _VALID_POLICIES:
            return explicit  # 大多數報表走此路徑

    # 2. 推導：從 job_lane 欄位的舊版備援
    job_lane = job.get("job_lane") or "interactive"
    if job_lane == "nightly":
        return POLICY_NIGHTLY_BOARD      # "nightly_board_refresh"
    return POLICY_INTERACTIVE            # "interactive_live_report"

NIGHTLY_POLICIES 目前為 frozenset，僅包含 nightly_board_refresh。當 nightly_alert_training_refresh 啟用後，它將被加入此集合，並在 run_nightly_job() 內部以不同方式路由。

NIGHTLY_POLICIES: frozenset[str] = frozenset({
    "nightly_board_refresh",
    "nightly_alert_training_refresh",   # 已定義但目前無 API 路由發出
})
注意事項 result_core_debug 是 JSONB 僅附加稽核 blob。鍵的順序不保證。請一律以鍵名讀取 execution_policy，不要依賴位置。不要在 nightly_alert_training_refresh 上建立邏輯 — 目前尚無 API 路由發出此政策。

7. Worker 排程器

夜間排程器是單一 Next.js API 路由，由 Railway 內建 cron 在約 UTC 午夜呼叫。這不是背景工作或佇列 — 它同步執行並插入 pricing_reports 列，Python Worker 再透過輪詢迴圈取得。

身份驗證

所有呼叫需要 Authorization: Bearer <INTERNAL_API_SECRET>。若環境變數缺失，路由回傳 500（而非 401）以使設定錯誤明顯可見。Token 無效 → 401。

去重複 — 23 小時視窗

插入前，路由查詢 listing_reports，檢查過去 23 小時內是否有已連結至該房源的排程任務（狀態：queued、running 或 ready）。若有，該房源以 duplicate_recent_ready 原因跳過。23 小時視窗（短於 24 小時）可容許 cron 抖動，同時不阻擋合法的重新執行。

強制略過（僅限本地開發） 傳送 { "listingId": "…", "force": true } 可跳過去重複檢查。僅在 WORKER_TARGET_ENV === "local" 時允許。在正式環境嘗試 force 會回傳 400。不帶房源篩選的 force 永遠被拒絕（防止意外的全量略過）。

合格門檻

僅有有效 airbnb.com/rooms/ URL（儲存在 input_attributes.listingUrl）的房源才會被排程。僅條件模式的房源以 missing_listing_url 或 invalid_listing_url 原因跳過。

時區感知日期範圍

每個房源的 listing_timezone 欄位（IANA 時區字串）用於計算正確的「本地明天」開始日期。這可防止 UTC+ 時區房源的 UTC 截斷錯誤。備援值：America/Los_Angeles。驗證透過 Intl.DateTimeFormat — 無效時區字串靜默回退。

// startDate = 房源時區的本地明天
// endDate   = 本地明天 + 29 天
// 每份夜間報表永遠涵蓋 30 晚

排程器不做的事

  • 不執行定價分析本身
  • 不重試失敗的任務
  • 不直接推送給 Worker — Worker 獨立輪詢
  • 不設定 completed_at 或 market_captured_at — 這些欄位由 Worker 擁有

回應格式

永遠回傳 HTTP 200（即使每個房源有個別錯誤），確保 Railway cron 不會因部分失敗而重試整個執行。回應本文：{ filter, force, scheduled, skipped, total, results[] }。

來源：src/app/api/internal/nightly/schedule/route.ts

9. 定價合約規範

唯一正式欄位

規則 — 由 docs/report-contract.md 強制執行 CalendarDay.recommendedDailyPrice 是唯一正式的每日建議定價。所有使用者端的價格呈現（儀表板看板磁磚、圖表建議系列、報表日曆、警示電子郵件）必須且只能從此欄位衍生。缺少此欄位的舊版報表則回退使用 basePrice。

10. 定價策略

定價流水線 — 逐日資料流

標準流水線（模式 A 與 B）

  1. 爬取 — 每個取樣日期的 Airbnb 搜尋結果。透過 collect_search_comps 進行雙晚搜尋（2 晚優先、1 晚備援）。
  2. 篩選 — 房源類型硬性門檻，接著是嚴格 / 中等 / 寬鬆容差層（詳見第 12 節）。
  3. 評分 — 對每個通過篩選的比較物件與目標房源進行相似度評分（詳見第 11 節）。
  4. 建議 — 高於門檻的前 K 名比較物件的相似度加權均值，含可選新房源折扣與價格合理性權重。
  5. baseDailyPrice = 該日期的相似度加權市場中位數。
  6. demandAdjustment = 週末加成 + 尖峰 / 活動旗標 + 市場緊俏度（詳見第 14 節）。鉗制範圍 0.90–1.05。
  7. recommendedDailyPrice = round(baseDailyPrice × demandAdjustment)。這是正式輸出。時間折扣排除在外。

基準流水線（模式 C）

當使用者提供偏好的可比房源 URL 時，該房源的基準優先引擎接管，結果再混合回標準日曆。完整細節見第 13 節。

插值

未直接取樣的日期由 interpolate_missing_days() 填充 — 在相鄰觀察值之間進行線性插值。當無相鄰觀察值時，插值日期回退使用視窗層級的整體中位數。

來源：worker/core/pricing_engine.py、worker/core/dynamic_pricing.py、worker/core/benchmark.py、docs/report-contract.md

11. 相似度計算方式

相似度是目標房源與每個候選比較物件之間計算的 0–1 加權分數。它同時決定篩選門檻與定價權重。

權重（V1 — 來自 worker/core/similarity.py）

最大總權重 = 3.0 + 2.8 + 2.2 + 1.8 + 1.2 + 1.0 = 12.0

1.8 容差 ±3

數值特徵公式

# 對每個容差為 `tol`、權重為 `w` 的數值特徵： score_component = max(0.0, 1.0 - abs(target - candidate) / tol) * w # 若任一值為 None（未知欄位）：部分分數 = 0.35 × w # 缺少欄位不會歸零總分 — 而是按比例降低。

房源類型類別公式

# 兩者已知 → 完全匹配 score += (1.0 if target.property_type == cand.property_type else 0.0) * 3.0 # 任一未知 → 部分分數（0.35 × 3.0 = 1.05）

設施重疊（Jaccard）

overlap = |target_amenities ∩ cand_amenities| / |target_amenities ∪ cand_amenities| score += overlap * 1.0 # 任一端為空 → 部分分數 0.35

最終分數

similarity = sum(score_components) / sum(weights) # 範圍：0.0 – 1.0

相似度門檻

任何 similarity < 0.40 的比較物件均被排除於定價權重之外。此門檻在篩選層之後、recommend_price() 被呼叫之前套用。被提升排名的比較物件（釘選 / 偏好比較物件使用 2× 加成）仍以其原始相似度分數接受門檻檢查。

釘選比較物件排名加成

effective_score = min(raw_similarity * 2.0, 0.98) # 僅用於排名順序 pricing_weight = raw_similarity # 定價永遠使用原始分數
重要區別 偏好 / 釘選比較物件被提升以在候選清單中排名靠前，增加進入定價前 K 名的機會。但其實際定價權重是原始（未加成）的相似度分數 — 加成分數絕不會膨脹定價。
來源：worker/core/similarity.py、worker/core/pricing_engine.py

12. 比較物件篩選方式

篩選發生在相似度評分之前。目標是在避免房源類型污染的同時，找到最大的結構一致比較物件集合。

房源類型硬性門檻

適用於所有層級 — 包含寬鬆層 此門檻阻擋與目標互斥的房源類型：
  • entire_home 目標 → 拒絕 private_room 與 shared_room 比較物件
  • private_room 目標 → 拒絕 entire_home 比較物件
  • 未知比較物件類型 → 允許（無法對無法讀取的資料施加懲罰）
  • 未知目標類型 → 不套用門檻

三層備援

層級容差範圍必要條件通過所需最低比較物件數
嚴格 accommodates ±2、bedrooms ±1、baths ±1、beds ±2 bedrooms 與 accommodates 必須非空 6
中等 accommodates ±3、bedrooms ±2、baths ±1.5 bedrooms 與 accommodates 必須非空 4
寬鬆 accommodates ±5、bedrooms ±3、baths ±2 無（允許 bedrooms / accommodates 缺失） 0（但若 0 個通過則 stage = insufficient_data）

解析順序：先嘗試嚴格 → 若少於 6 個比較物件，嘗試中等 → 若少於 4 個，落入寬鬆。寬鬆層不會將 0 個比較物件作為有效集合傳回呼叫端 — 它改為傳回 insufficient_data 階段，在上游觸發錯誤條件。

篩選後：相似度評分 + 門檻

通過篩選層的所有比較物件均進行相似度評分。原始分數 < 0.40 的比較物件，無論通過哪個篩選層，都會被排除於定價之外。篩選階段（嚴格 / 中等 / 寬鬆）存入 compsSummary.filterStage，並在報表透明度面板中顯示。

來源：worker/core/similarity.py → filter_similar_candidates()

13. 基準優先定價路徑

當使用者提供一或多個偏好的可比房源 URL（釘選基準）時，系統切換至兩階段定價模型，以基準房源的實際定價（而非市場中位數）錨定估算。

階段 1 — 基準錨定價格

Worker 直接導航至基準房源頁面（Playwright）並讀取每個取樣日期的預訂小工具夜間價格。這是主要且最可靠的價格信號。若 1 晚萃取失敗，以 2 晚最低入住備援重試（仍傳回每晚價格）。

若基準房源出現在 Airbnb 搜尋結果中，其價格則從搜尋卡片擷取（較高信心度，標記為 search_hit）。直接頁面萃取標記為 direct_page（較低信心度）。失敗 → failed。

階段 2 — 市場驗證 / 調整

從相同搜尋中收集較小的其他市場比較物件集合。其中位數成為市場參考價格。計算基準與市場之間的原始偏移量，鉗制至 ±25%，再依有效市場權重縮放後作為修正量套用。

# 核心混合公式 raw_offset = (market_median - benchmark_price) / benchmark_price capped_adj = clamp(raw_offset, -0.25, +0.25) final_price = benchmark_price × (1 + capped_adj × effective_weight) # 依基準抓取信心度的有效市場權重： # search_hit (高) → 0.30 （基準最可信） # direct_page (中) → 0.42 （有些不確定性） # direct_page (低) → 0.55 （本文萃取，最不可靠） # 額外調整： # - 市場比較物件少於 5 個 → 權重按比例縮小 # - 市場與基準差距 > 40% → 離群值保護，權重減半 # - 結構相似度（基準 vs 目標）< 0.45 → 權重 × 0.65 # - 結構相似度 0.45–0.70 → 權重 × 0.85

次要比較物件共識

可選的次要釘選基準 URL 被收集，其價格與主要基準和市場進行比較。共識信號（strong / mixed / divergent）透過小幅乘數進一步微調有效市場權重（0.90 支持 → 1.15 反對）。範圍刻意設計得很窄，使次要比較物件無法覆蓋主要信號。

透明度輸出

resultSummary.benchmarkInfo 中的基準區塊公開：基準 URL、平均價格、抓取狀態 / 方法、結構匹配分數 + 不匹配程度、市場偏移 %、有效權重、離群值旗標及每個次要比較物件的資料。

注意事項 — 基準設定 benchmark.py 中的基準常數無法透過環境變數設定（與夜間策略不同）。更改 BENCHMARK_MARKET_WEIGHT、BENCHMARK_MAX_ADJ 或信心度表格值需要程式碼部署。
來源：worker/core/benchmark.py、docs/report-contract.md §Benchmark Architecture

14. 動態需求與時間調整

每個日曆日均獲得一個 demandScore 及兩個乘數。需求調整用於產生 recommendedDailyPrice。時間乘數額外套用以產生 priceAfterTimeAdjustment — 這是內部流水線欄位，非正式建議定價。

需求評分公式

demand_score = clamp( 0.50 # 中性基準 + 0.20 × premium_index # 當日中位數 vs 前後 ±3 天 + 0.15 × tightness_index # (p75-p25)/中位數 — 低緊俏度 = 供應緊張 + weekend_boost # 五 / 六 = +0.08，日 = +0.04，平日 = 0 + flag_boost, # 尖峰 / 活動 = +0.15，低需求 = -0.15，否則 = 0 0.0, 1.0 )

需求調整（套用於 recommendedDailyPrice）

demand_adjustment = clamp(1.0 - (0.6 - demand_score) × 0.10, 0.90, 1.05) # 中性評分（0.6）→ 1.00（無變化） # 高需求（0.8+） → 1.02–1.05（輕微溢價） # 低需求（0.4） → 0.98–0.90（輕微折扣） # 範圍：baseDailyPrice 的 ×0.90 至 ×1.05

時間乘數（僅套用於 priceAfterTimeAdjustment — 臨時折扣）

compute_time_multiplier(today, target_date): days_away = target_date - today if days_away > 30: return 1.00 if days_away > 14: return 0.97 # -3% if days_away > 7: return 0.92 # -8% if days_away > 3: return 0.85 # -15% return 0.75 # 0–3 天：-25%

最終乘數（僅限內部）

final_multiplier = clamp(time_multiplier × demand_adjustment, 0.65, 1.05) priceAfterTimeAdjustment = baseDailyPrice × final_multiplier # recommendedDailyPrice = baseDailyPrice × demand_adjustment ← 正式欄位 # priceAfterTimeAdjustment 含時間折扣 ← 內部 / 舊版欄位
關鍵區別 recommendedDailyPrice 刻意不對臨時日期套用折扣。它代表房東應發布的建議定價。包含時間折扣的 priceAfterTimeAdjustment 是內部流水線階段；日曆中的 basePrice 因向後相容性而等於此值，供舊版 UI 元件使用。
來源：worker/core/dynamic_pricing.py、docs/report-contract.md

風險：listing_timezone 不總是有值 夜間排程器使用 listing_timezone 計算正確的「本地明天」日期範圍。缺少此欄位的房源回退至 America/Los_Angeles。UTC+ 時區但未填寫時區的房源可能收到從錯誤日曆日開始的日期範圍。

推論：src/core/pricingCore.ts 非正式定價 此 TypeScript 模組為 /r/demo 路由產生確定性的預設資料。它不連接 Python Worker 或任何即時 Airbnb 資料。新的 UI 元件不應在展示內容以外的任何情況下呼叫此模組。

22. 佇列與 Worker DB 機制

為何任務佇列建於 Postgres 而非獨立佇列

pricing_reports 同時扮演任務記錄與結果儲存的角色。這避免了分散式系統中常見的任務與結果分離問題，並讓 Worker 的任務認領具有與結果一致的事務保證。代價是必須以 FOR UPDATE SKIP LOCKED 防止競爭，以心跳機制偵測 Worker 崩潰。

原子性認領（claim_pricing_report）

-- 002 migration：初始版本
-- 012 migration：加入 job_lane + target_env 過濾

claim_pricing_report(
  p_worker_token  uuid,
  p_stale_minutes int  DEFAULT 15,
  p_job_lane      text DEFAULT 'interactive',
  p_target_env    text DEFAULT 'production'
)

-- 認領邏輯：
-- 1. 選取 status='queued' 或（status='running' 且 heartbeat 超過 p_stale_minutes）
-- 2. 依 job_lane + target_env 過濾（Worker 只認領自己的車道）
-- 3. ORDER BY created_at ASC（先進先出）
-- 4. LIMIT 1 FOR UPDATE SKIP LOCKED（防止並行 Worker 競爭）
-- 5. 原子性更新：status=running、worker_claimed_at、worker_claim_token、worker_attempts+1

心跳機制（heartbeat_pricing_report）

heartbeat_pricing_report(p_report_id uuid, p_worker_token uuid)
-- 僅在 id + worker_claim_token 匹配且 status='running' 時更新 worker_heartbeat_at
-- 傳回 boolean：true = 更新成功，false = token 不匹配（表示任務已被回收）

-- Worker 每 HEARTBEAT_SECONDS=10 秒呼叫一次
-- 主要迴圈每 POLL_SECONDS=5 秒輪詢
-- STALE_MINUTES=15：超過此時間無心跳的 running 任務可被重新認領

車道隔離（job_lane + target_env）

30. 模型訓練與驗證

訓練由 ml_sidecar/model.py::train_model() 執行，使用 XGBoost（reg:squarederror）在 log1p(price) 空間中訓練。

樣本權重

目前 ml_sidecar 的每筆訓練行 sample weight = recency_decay × comps_used_weight × priority_amenity_weight

  • Recency decay：exp(−0.03 × days_old)；越新的 observation 權重越高
  • Comparable density weight：1 + clip(comps_used, 0, 20) / 20；可比房源越多，樣本權重越高
  • Priority amenity weight：1 + (priority_amenity_ratio × 0.20)；冷氣 / 廚房 / 洗衣 / 烘衣訊號完整的樣本會被輕度上調
  • 最終 weight 會先 cap 在 5× 平均值，再正規化至 mean = 1
Current Behavior — 舊版 similarity/source-weight 公式已不適用 舊版文件中的 similarity_score × source_weight 屬於早期 research / legacy pipeline 概念；目前正式 ml_sidecar 訓練不再依賴這組欄位，而是直接使用 observation-based feature matrix 與上列三段權重。

交叉驗證策略

Current Behavior — 兩種 CV 模式 訓練前先計算 price_date 的唯一日期數與可用樣本量：
日期多樣性足夠：使用 TimeSeriesSplit，最多 5 folds，確保驗證遵守時間順序。
日期多樣性不足：退回 KFold(shuffle=False)，fold 數依樣本量自動落在 2–5 之間。
Risk — Degraded 驗證模式 當訓練資料日期多樣性不足時，CV 策略會自動降級為 KFold。此時 MAE / Q2 / R2 仍可作為方向性指標，但不能等同於 time-safe 的 out-of-time 驗證。

XGBoost 超參數（目前固定值）

objective = reg:squarederror
n_estimators = 600 (+ early_stopping_rounds=50 僅用於 fold 訓練)
max_depth = 5
learning_rate = 0.05
subsample = 0.85
colsample_bytree= 0.85
tree_method = hist
# 最終模型用全部訓練資料 fit，cross-val 結果只用於診斷

訓練結束後，模型用全部訓練資料做最終 fit，並輸出 feature importance、per-fold 指標，以及可解釋性相關產物。

現有限制

Risk — 新房源冷啟動問題 正規化路徑需要至少 7 天 × 3 表的觀察資料，才能通過閘門。新加入的房源觀察資料不足時，會退回 legacy fallback（pool_snapshot），訓練品質較低，特別是時間序列特徵的有效性受限。
Risk — runtime metadata 與本地 validation 仍是兩層 正式 report 目前只會帶當次執行的 summary.mlPricing；較完整的 batch / holdout / time-split 驗證仍是本地 ml/reports/ artifact。也就是說，badge 顯示的是這次 report 有沒有套到 ML，不等於 所有畫面都能共享同一份完整驗證 manifest。
Risk — Degraded CV 驗證 當訓練資料日期多樣性不足時，CV 降級為 KFold，指標可能過度樂觀。這不是 bug，是已知的資料量限制，程式碼有 warning 輸出。
